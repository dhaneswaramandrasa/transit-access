## Project Summary

**Transit Accessibility Index** — an open research tool mapping transit equity across Jabodetabek (Greater Jakarta). Users click any point on the map; the app resolves it to an H3 hexagon, returns equity scores, POI reachability, and nearby transit data, then generates an AI equity analysis via OpenRouter.

Academic context: Smart Society master's program, Hiroshima University. Phase 2 adds Hiroshima.

For full scoring methodology, read [methodology.md](./methodology.md).

---

## Current Project State (as of March 2026)

**The deployed app uses synthetically generated data.** The Python r5py routing pipeline (`src/`) defines the intended full-routing methodology but has not yet been run to completion. The web data files in `web/public/data/` are generated by Node.js scripts in `web/scripts/`.

When making changes:
- Changes to scoring logic must be applied to **both** `src/analysis/score_accessibility.py` (Python pipeline) **and** `web/scripts/generate-sample-data.mjs` (current synthetic data) to keep them in sync.
- The data schema must remain identical between both pipelines so the web app can consume either output without modification.

## H3 Spatial Unit

The core spatial unit is **H3 hexagons at resolution 8** (~0.74 km², ~461 m edge length).

The H3 resolution constant must be **identical in three places**:
1. `config/settings.yaml` → `analysis.h3_resolution: 8`
2. `web/scripts/generate-sample-data.mjs` → `const H3_RESOLUTION = 8`
3. `web/src/lib/store.ts` → Zustand default `h3Resolution: 8`

**If you change resolution: update all three locations and regenerate all data files.**

The UI also supports resolution 7 (district overview). Toggle between them via `h3Resolution` in the Zustand store. The map loads `jakarta_h3_scores_res7.geojson` when `h3Resolution === 7`.

## Equity Scoring Framework

Two independent scores per hex (see `methodology.md` for full formulas):

### Transit Need Score
```
Need = 0.25 × normPop + 0.25 × normDependent + 0.25 × normZeroVehicle + 0.25 × normInverseNJOP
```
All inputs min-max normalised to [0, 100] across all Jabodetabek hexes.

### Transit Accessibility Score
```
Access = 0.15 × walkBonus + 0.25 × normDistInv + 0.25 × normCapacity
       + 0.20 × normPOIDensity + 0.15 × normShedPOI
```

### Equity Gap
```
EquityGap = Need − Access
```

### Quadrant classification (median split)
```
transit-desert  → Need ≥ median AND Access < median
transit-ideal   → Need ≥ median AND Access ≥ median
car-suburb      → Need < median AND Access < median
over-served     → Need < median AND Access ≥ median
```

### Legacy composite score (kept for backward compatibility)
```
composite_score = mean(score_30min, score_60min)
percentile_rank = rank(composite_score) / n × 100
```
Both `composite_score` and `transit_need_score` / `transit_accessibility_score` / `equity_gap` / `quadrant` must be present in every GeoJSON feature.

## GeoJSON Feature Schema

Every feature in `jakarta_h3_scores.geojson` must have these properties:

```typescript
{
  h3_index: string;              // e.g. "88218adda3fffff"

  // Equity framework (primary)
  transit_need_score: number;          // 0–100
  transit_accessibility_score: number; // 0–100
  equity_gap: number;                  // −100 to +100
  quadrant: "transit-desert" | "transit-ideal" | "car-suburb" | "over-served";

  // Legacy composite (secondary)
  composite_score: number;        // 0–100
  score_30min: number;            // 0–100
  score_60min: number;            // 0–100
  percentile_rank: number;        // 0–100

  // POI counts
  hospital_30min: number;
  clinic_30min: number;
  market_30min: number;
  supermarket_30min: number;
  school_30min: number;
  park_30min: number;
  hospital_60min: number;
  clinic_60min: number;
  market_60min: number;
  supermarket_60min: number;
  school_60min: number;
  park_60min: number;

  // Demographics (demand side)
  pop_total: number;
  pct_dependent: number;          // % aged <15 or >65
  pct_zero_vehicle: number;       // % households, no car/motorcycle
  avg_njop: number;               // average land value proxy (IDR/m²)
  is_informal_settlement: boolean;

  // Transit supply
  dist_to_transit: number;        // km to nearest formal stop
  is_walkable_transit: boolean;   // dist_to_transit ≤ 0.8 km
  transit_capacity_weight: number; // 0–1
  local_poi_density: number;      // integer
  transit_shed_poi_count: number; // integer
}
```

## Boundary Modes

| Mode | Layer ID | Data File | Notes |
|---|---|---|---|
| `hex` | `h3-accessibility` | `jakarta_h3_scores.geojson` or `_res7` | H3 hexagons |
| `kelurahan` | `kelurahan-boundaries` | `jakarta_kelurahan_boundaries.geojson` | Only features with `hex_count > 0` rendered |
| `kecamatan` | `kecamatan-boundaries` | `jakarta_kecamatan_boundaries.geojson` | Only features with `hex_count > 0` rendered |

Kelurahan/kecamatan scores are area-weighted averages of their constituent H3 hexes. The spatial join uses a **vertex-based method** (centroid + 6 corner vertices tested against polygon) to avoid missing narrow administrative units.

## AI Analysis Endpoint (`POST /api/analyze`)

**Request body** (`AnalyzeRequest` in `route.ts`):
```typescript
{
  h3_index: string;
  h3_resolution: number;
  lat: number;
  lng: number;
  // Equity scores
  composite_score: number;
  score_30min: number;
  score_60min: number;
  transit_need_score: number;
  transit_accessibility_score: number;
  equity_gap: number;
  quadrant: string;
  // Legacy comparison stats
  jabodetabek_avg_score: number;
  jabodetabek_median_score: number;
  percentile_rank: number;
  threshold: 30 | 60;
  // Demographics
  pop_total: number;
  pct_dependent: number;
  pct_zero_vehicle: number;
  avg_njop: number;
  dist_to_transit: number;
  local_poi_density: number;
  transit_shed_poi_count: number;
  poi_counts: Record<string, number>;
  // Optional enrichment
  demographics?: { ... };
  transit_stops?: { transjakarta: number; krl: number; mrt: number; lrt: number; total: number };
}
```

**API**: OpenRouter (`https://openrouter.ai/api/v1/chat/completions`)
**Model**: `openai/gpt-oss-20b`
**Max tokens**: 400
**Streaming**: No (full response returned as `text/plain`)
**Env var**: `OPENROUTER_API_KEY`

The prompt asks the model to write **2 short paragraphs**:
1. Equity assessment — what does this quadrant classification mean for residents here?
2. One concrete policy recommendation grounded in Jabodetabek's transit context (TransJakarta, KRL, MRT, LRT, angkot)

## Map Layers (render order in `AccessibilityMap.tsx`)

1. `transit-corridors` — PathLayer, TransJakarta/KRL/MRT/LRT route lines (always visible)
2. `kelurahan-boundaries` or `kecamatan-boundaries` — GeoJsonLayer (when boundaryMode ≠ hex)
3. `h3-accessibility` — GeoJsonLayer (when boundaryMode === hex)
4. `transit-stops` — ScatterplotLayer (results phase only)
5. `poi-markers` — ScatterplotLayer (when reachablePOIs.length > 0)
6. `walking-route` — PathLayer (when a POI is selected)
7. `transit-walking-route` — PathLayer (when a transit stop is selected)
8. `click-marker` — ScatterplotLayer (pin at clicked coordinate)

**Click handling**: Layer `onClick` on `h3-accessibility`. Kelurahan/kecamatan clicks are caught by DeckGL root `onClick`. POI/transit-stop clicks have their own handlers. Coordinate-only clicks (no layer hit) use `latLngToCell` + `hexLookup.get()`.

## GeoJSON Lookup Performance

`AccessibilityMap.tsx` builds `Map<h3_index, HexProperties>` once on load (`buildHexLookup`). All click resolution is O(1). **Never iterate over features on every click.**

## Python Conventions

- Raw data → `data/raw/` only. Never modify files there.
- All processing outputs → `data/processed/`
- All paths from `config/settings.yaml` — never hardcode
- CRS storage: `EPSG:4326`; spatial calculations: `EPSG:32748` (UTM 48S)
- Use `pathlib.Path`, `logging` (not `print`), docstrings on all functions

---

## Web Conventions

- All global state in Zustand store (`lib/store.ts`) — no React Context for app state
- `AccessibilityMap.tsx` owns all deck.gl layer construction. Do not create layers elsewhere.
- Data files in `web/public/data/` are served as static assets — large enough that they must be cached (`Cache-Control: max-age=86400`)
- TypeScript: all component props and API request/response types must be explicitly typed
- Never call `latLngToCell` or iterate GeoJSON features inside render functions — do it in event handlers or effects

## Known Limitations (for code context)

- **No LRT GTFS** — LRT is represented in transit hub proximity model only; `transit_stops.lrt` will be 0 in most cases
- **No angkot data** — informal transit not in any data source; access is underestimated in kampung areas
- **Synthetic data in deployment** — scores are modeled, not routed; replace with Python pipeline output when complete
- **H3 hexes straddle physical barriers** — rivers, toll roads not respected by hex boundaries; routing handles this but visual attribution may feel arbitrary
- **AI analysis is generative** — label as insight, not authoritative planning data
- **Equal Need weights** — the 0.25/0.25/0.25/0.25 weighting is a starting baseline, not empirically calibrated
